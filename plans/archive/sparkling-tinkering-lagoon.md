### Fix A — Analytics chunk not loaded (CRITICAL)

**Root cause:** `'analytics'` is missing from `pageChunkMap` in `router.js`. The function `switchAnalyticsTab` exists in `handlers-sales-orders.js` (line 4828) → builds into `chunk-sales.js`. Since analytics has no chunk mapping, navigating to `#analytics` never loads the sales chunk. All 11 `onclick="handlers.switchAnalyticsTab(...)"` calls throw TypeError.

**Complication:** Analytics calls handlers from 3 chunks, but the router only loads 1 chunk per route (no multi-chunk mechanism — confirmed in router.js lines 259–271):
| Chunk | Handlers needed | Count |
|-------|----------------|-------|
| `chunk-sales.js` | switchAnalyticsTab, showWeeklyReport, showAnalyticsCustomization, toggleAnalyticsCompare, switchChartType, exportAnalyticsCSV, setAnalyticsPeriod, refreshAnalytics, showGoalSettings, showDetailedAnalytics + 3 more | 13 |
| `chunk-inventory.js` | showSalesVelocity | 1 |
| `chunk-settings.js` | showCustomMetricBuilder, showAnalyticsDigestSettings | 2 |

**Fix:**
1. Add `'analytics': 'sales'` to `pageChunkMap` in router.js (~line 55) — loads 13/16 handlers
2. Move 3 outlier handlers into `handlers-core.js` (so they're in core-bundle)
3. Remove those 3 from `split-deferred-chunks.js` HANDLERS_CHUNK_MAP AND from their source chunk files to prevent double-definition

**Files to edit:**
| File | Change | Risk |
|------|--------|------|
| `src/frontend/core/router.js` ~line 55 | Add `'analytics': 'sales',` | Tiny — 1 line |
| `src/frontend/handlers/handlers-core.js` | Add showSalesVelocity (~80 LOC), showCustomMetricBuilder (~85 LOC), showAnalyticsDigestSettings (~60 LOC) | Medium — must copy exact implementations |
| `scripts/split-deferred-chunks.js` | Remove 3 entries from HANDLERS_CHUNK_MAP | Small |
| `src/frontend/handlers/handlers-deferred.js` | Remove 3 handler bodies at exact lines: showSalesVelocity (10648–10760), showCustomMetricBuilder (25114–25198), showAnalyticsDigestSettings (25239–25283). SOURCE OF TRUTH — chunk files are auto-generated. |
| `scripts/split-deferred-chunks.js` | Remove 3 entries from HANDLERS_CHUNK_MAP | Small |
| Then: `bun scripts/split-deferred-chunks.js` | Regenerates chunk files without removed handlers |
| Then: `bun scripts/build-dev-bundle.js` | Regenerates core-bundle.js with new core handlers |

**DO NOT** edit `handlers-inventory-catalog.js` or `handlers-settings-account.js` directly — they are auto-generated by `split-deferred-chunks.js` from `handlers-deferred.js`.

**Dependency check (verified):** All 3 handlers use only core-bundle utilities (`store`, `modals`, `components`, `escapeHtml`). They reference `handlers.showPriceDropSuggestion`, `handlers.deleteCustomMetric`, `handlers.saveCustomMetric`, `handlers.saveDigestSettings` via HTML `onclick` strings (not direct JS calls) — these stay in deferred and resolve at click time. No broken dependencies.

**Effort:** Medium (~230 LOC added to core, ~230 LOC removed from deferred, 4 files + 2 script runs)

**Risks:**
- Double-definition: If entries not removed from both `handlers-deferred.js` AND `split-deferred-chunks.js` HANDLERS_CHUNK_MAP, re-running the splitter regenerates chunk files with the old handler bodies. Always update both.
- Version hash: `build-dev-bundle.js` auto-computes hash and injects into `router.js`'s `const v`. Don't change that line's quote style (single quotes).

**Route alias side effect (verified round 5):** `#predictions`, `#market-intel`, `#suppliers`, `#report-builder` alias to `#analytics` (aliases resolve BEFORE chunk loading — router.js lines 199-204). After Fix A, these routes load the `sales` chunk via the analytics mapping. The `intelligence` chunk is NOT loaded (alias redirects before chunk lookup). This is pre-existing (intelligence wasn't loading before either) — not a regression. Intelligence-specific handlers remain unavailable.

**Impact:** ~15 items → Pass (#340, #171, #174, #176-190)

### Fix C — Ad-hoc modals bypass modal system (MEDIUM)

**Root cause:** 3 places create modals by appending `div.modal-overlay` to `document.body`, bypassing `modals.show()`. This means no Escape handler, no ARIA attributes, no focus trap, no focus restore:
1. `handlers-core.js` line 247 — `setMonthlyGoal` uses `document.body.appendChild(modal)`
2. `widgets.js` line 1421 — `tablePrefs.showColumnPicker` uses `document.body.appendChild(modal)`
3. `handlers-core.js` line 252 — confirmDialog inside setMonthlyGoal (NOTE: may not be a separate function — needs verification during implementation)

**Fix:** Refactor all 3 to use `modals.show(html)` which already provides Escape, ARIA, focus trap, focus restore. Must also update their close/save callbacks: change `this.closest('.modal-overlay').remove()` → `modals.close()`. The `applyColumnPicker` function (widgets.js line 1424) calls `modal.closest('.modal-overlay').remove()` — must change to `modals.close()`.

**Files:**
| File | Change |
|------|--------|
| `src/frontend/handlers/handlers-core.js` lines 244-265 | Refactor setMonthlyGoal to use modals.show(); update saveMonthlyGoal cleanup |
| `src/frontend/ui/widgets.js` lines 1395-1430 | Refactor showColumnPicker to use modals.show(); update applyColumnPicker |

**Effort:** Small (~25 LOC changed across 2 files)

**Risks:**
- `modals.show()` writes to `#modal-container`, not `document.body`. Old code used `querySelector('.modal-overlay')` to find and remove — that selector would miss the modal in `#modal-container`. Must update all cleanup paths.
- `showColumnPicker` uses a callback parameter `onApply` but currently ignores it (calls `renderCurrentPage()` instead). This behavior must be preserved.
- No stacked modal risk: `modals.show()` replaces `#modal-container` innerHTML, preventing stacking.

**Impact:** Resolves #294 (Fail/Medium — Escape). Also pre-enables Fix E's ARIA benefits for these 3 modals.

### Fix D — Sidebar responsive CSS/JS mismatch (HIGH)

**Root cause (3 disconnects):**
1. `html.desktop-lock` CSS (main.css lines 2176-2194) includes `.menu-button { display: none !important }` — hides the hamburger that exists in components.js line 309. No JS ever removes `desktop-lock`, so hamburger is permanently hidden.
2. 768px CSS expects `.sidebar.mobile-open` (line 60629) but JS only toggles `.open` (components.js line 309) — dead CSS rule.
3. `.mobile-header` and `.mobile-menu-btn` CSS exist (lines 60661, 60678) but no HTML template renders them.

**Fix:**
1. Remove or condition the `.menu-button { display: none !important }` inside `html.desktop-lock` (line 2182-2183)
2. Change `.sidebar.mobile-open` → `.sidebar.open` at line 60629
3. Add `.mobile-header` template to components.js for ≤768px

**Tension:** `desktop-lock` is a zoom-based layout freeze (applied when `outerWidth < 1100px`). Removing its `.menu-button` override may reintroduce zoom-jank on reduced viewports. The fix should be scoped: only remove the `.menu-button` hide, not the entire `desktop-lock` block.

**Dead zone:** 1024-1100px range where desktop-lock is active but sidebar CSS breakpoint hasn't fired. The sidebar is locked visible but content may be constrained. This is a known trade-off — not addressed in this plan.

**Files:**
| File | Change |
|------|--------|
| `src/frontend/styles/main.css` line 2182-2183 | Remove `.menu-button { display: none !important }` from desktop-lock |
| `src/frontend/styles/main.css` line 60629 | `.mobile-open` → `.open` |
| `src/frontend/ui/components.js` | Add mobile-header template (~25 LOC) |

**Effort:** Small (~30 LOC)

**Risk:** Highest visual regression risk of all 5 fixes. Test at multiple viewport widths (1200, 1024, 900, 768, 640).

**Impact:** Resolves #314 (Fail/High), #313 (Fail/High), #30, #31

### Fix E — ARIA dropdowns + inert (HIGH)

**Root cause:**
1. Shop-switch button (components.js line 233) and notifications button (line 322) missing `aria-haspopup` — the MutationObserver at app.js lines 7715-7727 only syncs `aria-expanded` on elements that already have `aria-haspopup`
2. No `inert` on `#main-content` when modal open — screen readers can arrow-key outside modal
3. Ad-hoc modals missing all ARIA → **already solved by Fix C refactor**

**Fix:**
1. Add `aria-haspopup="true"` to 2 buttons in components.js
2. In `modals.show()` (~line 20): add `document.getElementById('main-content')?.setAttribute('inert', '')`
3. In `modals.close()` (~line 63): add `document.getElementById('main-content')?.removeAttribute('inert')` BEFORE focus restore (order matters — element must be interactive before focus moves to it)

**Known gap after fix:** `modals.confirm()` (modals.js lines 90-126) bypasses `modals.show()` entirely — writes directly to `#modal-container.innerHTML` with its own button onclick handlers. It will NOT apply inert or register Escape handler. Options: (a) add inert toggle inside `confirm()` too (~4 more LOC), or (b) refactor `confirm()` to call `show()` internally. Option (a) is simpler and lower risk — recommend including it in Fix E.

**Files:**
| File | Change |
|------|--------|
| `src/frontend/ui/components.js` lines 233, 322 | Add `aria-haspopup="true"` (2 LOC) |
| `src/frontend/ui/modals.js` lines ~20, ~63 | Add/remove inert in `show()` and `close()` (4 LOC) |
| `src/frontend/ui/modals.js` lines ~90-126 | Add/remove inert in `confirm()` too (4 LOC) |

**Effort:** Tiny (~10 LOC)

**Risk:**
- If navigation occurs while modal is open, `renderApp()` replaces `#main-content` DOM — inert attribute lost from new element. Old `modals.close()` listeners still point to replaced DOM. Low probability in normal use.
- Inert removal MUST happen before focus restore in `close()` — wrong order → focus fails on still-inert element.

**Impact:** Resolves #278 (Fail/High), #328 (Fail/High), #320, #276, #277

- SW cache: after rebuild, old SW may serve stale assets. Unregister SW or hard refresh between tests
