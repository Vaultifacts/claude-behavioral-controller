**Step 1: Add `storeKey` to alias entries (router.js:270-289)**

```js
routeAliases: {
    'my-listings': { target: 'listings', tab: null },
    'orders': { target: 'orders-sales', tab: 'orders' },
    'sales': { target: 'orders-sales', tab: 'sales-summary' },

    'market-intel': { target: 'analytics', tab: 'market-intel', storeKey: 'analyticsTab' },
    'suppliers': { target: 'analytics', tab: 'sourcing', storeKey: 'analyticsTab' },
    'platform-health': { target: 'shops', tab: 'health' },

    'terms-of-service': { target: 'help-support', tab: 'terms' },
    'privacy-policy': { target: 'help-support', tab: 'privacy' },
},
```

Only 7 entries get `storeKey` (the ones targeting pages with verified tab store keys).
Entries without `storeKey` still redirect but don't try to set a nonexistent tab — safe.

**Step 2: Fix handler (router.js:299-303)**

Current (broken):
```js
if (alias) {
    path = alias.target;
    store.setState({ activeTab: alias.tab });
    window.history.replaceState({}, '', `#${path}`);
}
```

Fix:
```js
if (alias) {
    path = alias.target;
    if (alias.storeKey && alias.tab) {
        store.setState({ [alias.storeKey]: alias.tab });
    }
    window.history.replaceState({}, '', `#${path}`);
}
```

**Why this is better than a separate tabKeyMap:**
- Data locality: each alias carries its own store key — no separate lookup table
- Simpler handler: 3 lines instead of 10
- Self-documenting: reading the alias map tells you everything about what it does
- Easier to maintain: adding a new alias with tab support = add `storeKey` to that entry

## P5 — UX bugs

### 5.1: Changelog search loses focus on each keystroke

**Previous claim "NO event handlers" was WRONG.** The search input at `pages-community-help.js:2148`
DOES have `oninput="handlers.searchChangelog(this.value)"`. The handler exists at
`handlers-community-help.js:339`:
```js
searchChangelog: function(query) {
    store.setState({ changelogSearch: query });
    renderApp(pages.changelog());
},
```

**Real bug:** Full-page `renderApp()` on every keystroke destroys and recreates the input element,
losing cursor position and focus. Typing is effectively impossible.

**Files (NOT in core bundle):**
- `src/frontend/handlers/handlers-community-help.js:339`
- `src/frontend/handlers/handlers-deferred.js` (duplicated handler)

**Fix:** After re-render, restore focus and cursor position:
```js
searchChangelog: function(query) {
    store.setState({ changelogSearch: query });
    renderApp(pages.changelog());
    const input = document.querySelector('[aria-label="Search changelog"]');
    if (input) {
        input.focus();
        input.selectionStart = input.selectionEnd = input.value.length;
    }
},
```

**Better long-term fix:** Filter DOM nodes via CSS `display:none` instead of full re-render. But
the focus-restore approach is a minimal fix that matches existing patterns in the codebase.

### 5.2: Shortcuts modal shows wrong content (parameter swap)

**Previous claim "raw HTML in shortcuts array" was WRONG.** The shortcuts array at
`pages-settings-account.js:713-719` is fine. The bug is in the handler.

**Root cause:** `handlers-settings-account.js:781` calls:
```js
modals.show('Keyboard Shortcuts', `<div style="padding: 8px;">...</div>`);
```
But `modals.show(content, sizeClass)` takes **content as the first arg** and **CSS size class as second**.
So `'Keyboard Shortcuts'` is rendered as the modal body, and the HTML is used as a CSS class string.

**Files (NOT in core bundle):**
- `src/frontend/handlers/handlers-settings-account.js:781`
- `src/frontend/handlers/handlers-deferred.js:5845` (duplicated handler)

**Fix:** Remove the first argument (title string) and pass only the HTML content:
```js
modals.show(`<div style="padding: 8px;">
    <h3 style="...">Keyboard Shortcuts</h3>
    ...
</div>`);
```
Or if `modals.show` supports a title option, use that instead. Verify the `modals.show()` signature
at `modals.js:13` before implementing.
